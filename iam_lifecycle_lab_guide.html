User Lifecycle Management (Provisioning & Deprovisioning) Lab

Description

In this project I created a User Lifecycle Management home lab using Oracle VirtualBox, Windows Server 2022, Active Directory, and a Windows 11 client machine. This lab helped me build hands-on experience with core IAM concepts such as domain setup, DNS, organizational units, security groups, user provisioning, RBAC-based access assignments, domain authentication, and user deprovisioning. I also automated onboarding and offboarding tasks using PowerShell scripts and CSV files to simulate how joiner and leaver workflows can be handled in a real-world identity and access management environment.

Languages and Utilities Used

  • PowerShell
  • Oracle VirtualBox
  • Active Directory Users and Computers
  • Windows Server Manager
  • CSV files for onboarding and offboarding simulation

Environments Used

  • Windows Server 2022 Standard Evaluation
  • Windows 11 Pro
  • Oracle VirtualBox

Project Goals

  • Build a working Active Directory domain from scratch
  • Create a Windows 11 client and join it to the domain
  • Design organizational units and security groups for role-based access control
  • Automate user provisioning from a CSV file with PowerShell
  • Automate user deprovisioning by disabling accounts, removing group access, and moving accounts to a DisabledUsers OU
  • Document logging output for audit visibility

Lab Architecture

DC01 / DC02 (Windows Server 2022) |- Active Directory Domain Services |- DNS |- OUs and Security Groups |- Provisioning & Deprovisioning Scripts CL01 (Windows 11 Pro) |- Joined to lab.local domain |- Tested domain logon as provisioned user PowerShell Automation |- new_users.csv -> Provision-Users.ps1 |- leavers.csv -> Leaver-Users.ps1

Step 1: Install the Windows Server VM

I created a Windows Server 2022 virtual machine in Oracle VirtualBox and configured the base resources for the domain controller. I used the Windows Server 2022 Standard Evaluation ISO and completed the installation using the Desktop Experience option.

Step 2: Install the Windows 11 Client VM

I created a separate Windows 11 client machine in VirtualBox to simulate an end user workstation. During the setup, I selected the clean installation path and used the Windows 11 setup process inside the VM.

Step 3: Bypass Windows 11 Virtual Hardware Checks

Because the lab was built in VirtualBox, I bypassed Windows 11 TPM and Secure Boot requirements by opening the registry editor during setup and creating a LabConfig key under the Setup registry path.

HKEY_LOCAL_MACHINE\SYSTEM\Setup\LabConfig BypassTPMCheck = 1 BypassSecureBootCheck = 1 BypassRAMCheck = 1

Step 4: Promote the Server to a Domain Controller

After installing Active Directory Domain Services, I promoted the Windows Server machine to a domain controller and created a new forest using the domain name lab.local. This established the foundation of the IAM lab.

The domain used in this project was lab.local with a NetBIOS name of LAB.

Step 5: Build the Active Directory Structure

I created organizational units and security groups to support a role-based access control model. This allowed me to separate users, disabled accounts, workstations, and group assignments in a way that closely mirrors a real enterprise identity environment.

  • Users
  • DisabledUsers
  • Groups
  • Workstations
  • ServiceAccounts

Sample security groups created in the lab:

  • GRP_Finance_Read
  • GRP_HR_Read
  • GRP_Trading_AppUsers
  • GRP_VPN_Users

Step 6: Join the Windows 11 Client to the Domain

I configured DNS on the Windows 11 client to point to the domain controller and then joined the workstation to the lab.local domain. After the reboot, I verified that the login screen displayed the domain sign-in target.

Step 7: Create the Provisioning CSV File

To simulate an HR onboarding feed, I created a CSV file containing first name, last name, username, department, and role. This file was used as the input for my provisioning script.

FirstName,LastName,Username,Department,Role Mike,Ross,mross,HR,Coordinator Havery,Spector,hspecy,Trading,Trader Huey,Freeman,hfreeman,Finance,Analyst

Step 8: Create the PowerShell Provisioning Script

I wrote a PowerShell script that imported the CSV, created new Active Directory users, applied default passwords, enforced password changes at next logon, and added users to security groups based on department.

I initially received a Directory object not found error because the target path in the script used an OU path that did not match the actual AD location. I corrected the path and successfully reran the script.

Step 9: Configure PowerShell Execution Policy and Run the Provisioning Script

I changed the PowerShell execution policy to allow locally created scripts to run, then launched the provisioning script from my IAMLab folder.

Set-ExecutionPolicy RemoteSigned cd C:\Users\Administrator\IAMLab .\Provision-Users.ps1

Step 10: Verify Group Membership and Logging Output

After the script completed, I validated group membership in Active Directory and reviewed the provisioning log file to confirm successful user creation and RBAC-based group assignments.

Step 11: Create the Leaver CSV File

To simulate terminated users, I created a second CSV file listing usernames, ticket numbers, and business reasons for deprovisioning.

Username,Ticket,Reason mross,REQ-2001,Termination hspecy,REQ-2002,Voluntary resignation

Step 12: Create and Run the Leaver Script

I built a PowerShell deprovisioning script that removed users from access groups, disabled their accounts, updated the description with ticket details, and moved the accounts into the DisabledUsers OU.

cd C:\Users\Administrator\IAMLab .\Leaver-Users.ps1
The target path for disabled accounts was confirmed as OU=DisabledUsers,DC=lab,DC=local.

Step 13: Verify Deprovisioning Results

I verified that the terminated user accounts were moved into the DisabledUsers OU, showed as disabled in Active Directory, and retained only the default Domain Users membership where appropriate.

Lessons Learned

One of the biggest lessons from this project was how important exact Active Directory paths are when automating user creation and deprovisioning. I also strengthened my understanding of DNS dependency during domain authentication, PowerShell execution policies, and how IAM tasks can be modeled with repeatable scripts instead of fully manual actions. Working through the errors in the scripts and validating the outcomes in Active Directory made this project much more practical and realistic.

Conclusion

This lab gave me practical experience building a small IAM environment from the ground up. By the end of the project, I successfully created a joiner and leaver workflow using Active Directory and PowerShell. This helped me better understand how user lifecycle management works in enterprise environments and gave me a strong foundation for future IAM, cybersecurity, and identity engineering projects.
